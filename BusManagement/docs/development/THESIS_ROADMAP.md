## 1. Thesis Vision

The project is evolving from a simple Bus Management System into a **thesis-level Bus Operations & Decision Support System**. The system is built around three pillars, in order of priority:

1. **Operations** (primary focus, ~70-80%) — Fleet Management, Driver Management, Trip Scheduling, Dispatch Center, Business Rule Validation, Incident Management, Analytics.
2. **Intelligent Decision Support** (~20-30%) — Demand Forecast, Recommendation Engine, Vehicle Replacement Recommendation, Driver Recommendation, Cost/Revenue Estimation, What-if Simulation.
3. **Customer Portal** (minimal scope, ~10-20%) — Search Trips, View Seat Availability, Simple Booking, Booking History. Goal is to prove the data flow, not build a full e-commerce experience.

## 2. Project Scope

- Backend: Java / Spring Boot 4, Spring Data JPA / Hibernate, MySQL, Thymeleaf, Spring Security (currently disabled/permit-all by design).
- The existing codebase (Fleet, Trip FSM, AI auto-assign, Dashboard & Analytics) is the **foundation**, not something to be rewritten. All new work builds on top of it.
- The system currently contains **rule-based automation**, not real AI. This distinction must be preserved and stated accurately in all documentation and in the thesis defense.

## 3. Architectural Principles (must be respected in every phase)

- Reuse existing `TripService` logic whenever possible. Do not duplicate business logic that already exists (e.g. `findBestAvailableBus()`, `findBestAvailableDriver()`, `validateBusForTrip()`, `validateStaffForTrip()`, `getDrivingHoursForDate()`).
- Do not create parallel architecture. New modules follow the existing layered pattern: Controller → Service → Repository → Domain.
- New admin controllers must follow the `AdminBusController` pattern (inject Service only) — **not** the `AdminTripManagementController` pattern (injects Repository directly alongside Service; this is a known, documented anti-pattern, not something to copy into new code).
- **Business Rule Validation** and **AI Prediction (Demand Forecast)** are two independent components. Business Rule Validation verifies operational constraints; AI Prediction forecasts future demand. Decision Support consumes outputs from both — neither is allowed to silently absorb the other's responsibility.
- **Recommendation Engine is an orchestrator, not a prediction engine.** It composes outputs from Demand Forecast, Resource Selection (which reuses existing rule+availability-filtered methods), Cost Estimation, and a final Business Rule Validation re-confirmation gate. It does not contain its own prediction or rule logic.
- "Business Rules" and "Resource Availability" are not separate sequential services in the implementation — in the existing code (`findBestAvailableBus()`) they are interleaved filters over the same candidate stream. Keep them as two conceptual boxes in diagrams/thesis writing, but do not implement them as two duplicated service layers.
- The final Business Rule Validation gate inside the Recommendation Engine pipeline reuses the same throw-based validators (`validateBusForTrip`/`validateStaffForTrip`) via a non-throwing dry-run wrapper — mirroring the existing double-check pattern already used in `TripService.confirmAutoAssignedTrip()`.
- Incident and Booking are independent entities referencing `Trip` via foreign key. Do not add their responsibilities as new fields directly on `Trip` — `Trip` already carries FSM + soft-delete + self-reference responsibilities and must not become a god-entity.
- What-if Simulation must stay at the aggregate/statistical level. Do not refactor `findBestAvailableBus()`/`findBestAvailableDriver()` into pure functions accepting hypothetical resource pools — this is the single highest refactoring-risk change identified in the whole roadmap and is explicitly rejected in favor of a narrower, statistics-based simulation.
- Every phase must leave the application in a runnable, independently testable state. No phase may be left half-implemented across a session boundary without being marked `IN PROGRESS` with clear notes.
- Minimize refactoring. Prefer additive changes (new entity, new service, new controller) over modifying existing, already-verified logic (especially anything inside `TripService`).

## 4. Non-Goals (explicitly out of scope unless separately approved)

- No real ML/deep learning model. Demand Forecast uses simple statistical methods (moving average / linear regression) written in plain Java — no new ML dependency.
- No full e-commerce features for Customer Portal (no wallet, voucher, loyalty points, online payment gateway).
- No seat-map / specific seat number selection — only aggregate seat availability (`totalSeats - ticketsSold`).
- No enabling of real Spring Security / RBAC login as part of this roadmap. `SecurityConfig` remains permit-all unless the project owner explicitly requests turning it on as a separate, standalone task.
- No notification system (email/SMS) unless separately requested (dependency `spring-boot-starter-mail` exists in `pom.xml` but is unused by design, for now).
- No deep What-if Simulation that re-runs the full resource-assignment engine against hypothetical inputs (see architectural principles above).
- **No driver-facing portal** (original proposal section II.1–II.4: personal schedule view, QR passenger check-in, driver-side status updates, driving-hours logbook). All of these presuppose an authenticated driver session, and enabling login is itself out of scope (see above). The only part of section II carried into this roadmap is **II.5 incident reporting — deliberately re-interpreted as an Admin-recorded CRUD** (Phase 2): the Admin records incidents on behalf of drivers, because without authentication the reporter cannot be a driver session. State this re-interpretation explicitly in the thesis defense.
- **No fare/pricing management module** (original proposal I.1.d: price by route / bus type / time slot). `Trip.price` stays a manually-entered per-trip field; Cost/Revenue Estimation (Phase 7) consumes it as-is.
- **No ticket cancellation and no customer reviews/feedback** (original proposal III.3.2, III.4) — Phase 9's Booking flow is create + view history only, per the "prove the data flow, not e-commerce" pillar weighting.
- **No daily cron job** (original proposal IV.3: nightly reset of driving hours / maintenance schedule updates). `totalDrivingHours24h` remains documented mock-seed data (see Developer Notes); adding a scheduler is a separate owner decision.

## 5. Roadmap (approved — do not reorder, redesign, or simplify without explicit owner approval)

### Phase 0 — Foundation
- **Objective:** Unblock historical data persistence; this is the prerequisite for the entire Decision Support pillar.
- **Scope:** Introduce a persistent profile (schema not wiped on every restart — currently `ddl-auto=create-drop` + `DataInitializer.run()` deletes and reseeds all tables on every startup); add `Trip.createdAt`.
- **Dependencies:** None.
- **Expected deliverables:** A working persistent profile (dev/demo), `Trip.createdAt` column added, existing flows unaffected.
- **Refactoring risk:** None.
- **Current status:** COMPLETED (2026-07-16)
- **As delivered:**
  - Default profile is now **persistent** (`ddl-auto=update`); wiping/reseeding requires explicitly opting in via `-Dspring-boot.run.profiles=demo` (`application-demo.properties`, `ddl-auto=create-drop`). Owner-approved decision: inverting the default so an accidental restart can never destroy the historical dataset.
  - `DataInitializer` gated with `@Profile("demo")` — this is the actual fix; changing `ddl-auto` alone was insufficient because the bean calls `deleteAll()` on every table regardless of DDL policy.
  - `Trip.createdAt` added via Hibernate `@CreationTimestamp` (column `created_at`, nullable, `updatable = false`).
  - **Scope addition approved during the phase:** `src/test/resources/application.properties` added to isolate tests onto a separate `busmanagement_test` database (auto-created via `createDatabaseIfNotExist=true`). Discovered during analysis that `mvnw test` was wiping the real database on every run — same root cause as the phase objective, so it was folded in with owner approval.

### Phase 3 — Business Rule Validation as an Independent Component
- **Objective:** Make the existing validation logic consumable by Decision Support without throwing exceptions.
- **Scope:** Add non-throwing dry-run wrapper methods around the existing `validateBusForTrip()` / `validateStaffForTrip()` (return a structured result instead of throwing). Do **not** rewrite the underlying validation logic.
- **Dependencies:** None.
- **Expected deliverables:** New dry-run methods in `TripService`, existing throw-based flows unchanged and unaffected.
- **Refactoring risk:** Low if implemented strictly as a wrapper; would be high if the underlying logic were rewritten (explicitly disallowed).
- **Current status:** COMPLETED (2026-07-20)
- **As delivered:**
  - `service/ValidationResult.java` — immutable result object (`valid` / `failureReason` / `warning`), private constructor + static factories `pass()` / `pass(warning)` / `fail(reason)`, following the `AutoAssignResult` precedent in the same package.
  - `TripService.validateBusForTripDryRun(Bus, Trip, Long)` and `TripService.validateStaffForTripDryRun(Trip, Long)` — public, four-line `try`/`catch` wrappers delegating to the untouched private validators. **Zero changes to existing methods**; the two validators and all four throw-based callers are byte-for-byte unchanged.
  - Owner-approved: **pure catch-wrapper, fail-fast preserved** — a `ValidationResult` carries **at most one** failure reason, being the first rule violated. Sufficient for Phase 7's gate, whose responsibility is only pass/fail confirmation. Multi-error reporting is explicitly deferred as a separate architectural enhancement outside Phase 3, because it would require rewriting the validators.
  - Owner-approved by extension of the same principle: the wrappers catch **only `IllegalArgumentException`** (the type the validators use for business-rule violations). Technical faults propagate — see Developer Notes.
  - `validateBusForTrip()`'s existing `String` warning channel (currently always `null`) is forwarded into `ValidationResult.warning` rather than discarded.
  - New: `src/test/java/.../service/TripServiceValidationDryRunTest.java` (6 tests). Phase 3 has no UI, so the project's drive-the-app convention cannot reach these methods — see Session Log for how both were verified.

- **Dependencies:** Phase 3 (optional reuse).
- **Expected deliverables:** Driver Recommendation view/report (under-utilization/availability focus, per the scope above).
- **Refactoring risk:** None.
- **Current status:** COMPLETED (2026-07-20)

### Phase 5 — Historical Data Preparation
- **Objective:** Build the dataset that Demand Forecast will consume. Deliberately kept as its own phase, separate from the forecasting algorithm, so the thesis can present the "data" concern and the "algorithm" concern distinctly.
- **Scope:** Backfill/demo-data script generating several weeks of synthetic historical trips (COMPLETED/CANCELLED, varied by route/time slot) — necessary because real historical data will not accumulate fast enough during the thesis timeline. The dataset's **time axis is `Trip.departureTime`** (freely settable to past dates; every existing time-window query in `TripRepository` already keys on it). **Do not attempt to backdate `Trip.createdAt`** — `@CreationTimestamp` stamps INSERT time unconditionally and the column is `updatable = false`, so backfilled rows will carry the script's run date there; that is expected and harmless (see Developer Notes).
- **Dependencies:** Phase 0.
- **Expected deliverables:** A reusable backfill script/utility and a populated historical dataset in the persistent profile.
- **Refactoring risk:** None.
- **Current status:** COMPLETED (2026-07-20)
- **As delivered:**
  - `config/HistoricalDataBackfill` — a `CommandLineRunner` gated `@Profile("backfill")`, mirroring `DataInitializer`'s `@Profile("demo")` pattern. Run with `./mvnw spring-boot:run -Dspring-boot.run.profiles=backfill`. That profile does **not** activate `demo`, so nothing is wiped.
  - One additive derived query, `TripRepository.existsByRouteIdAndDepartureTime(...)`. No other existing file changed.
  - **Dataset:** 1,260 trips = 5 routes × 3 departure slots (06:00 / 12:00 / 18:00) × 84 days, ending yesterday. 1,183 `COMPLETED` + 77 `CANCELLED` (~6%). Time axis is `departureTime`, exactly as scoped.
  - **Exactly idempotent, with no new column on `Trip`:** every row is a deterministic `(route, departureTime)` slot, skipped if already present. Proven — the second run created 0 and skipped 1,260.
  - **Deterministic:** fixed seed (`20260720`), so the same dataset regenerates anywhere.
  - Owner-approved by scope: writes via repository rather than `TripService`, because `createManualTrip()` validates against *current* bus status and odometer, which is meaningless for rows dated three months back. `DataInitializer` sets the precedent that seeding bypasses services. Business-rule consistency is instead guaranteed **by construction** — see Developer Notes.
  - `Trip.createdAt` deliberately **not** backdated, per the scope line and Developer Notes.
  - **Odometer sync (added 2026-07-20 after the consistency re-check, owner-approved):** completed trips add their route distance to `Bus.odometer` **and the same amount to `lastMaintenanceOdometer`**, leaving `kmSinceLastMaintenance` — and therefore every maintenance rule — untouched. Without this the dataset and the fleet's mileage contradicted each other and Phase 7 had no real wear signal. See Hidden Cost #7 (resolved) and Developer Notes.

  - **Owner-approved deviation from this scope line (2026-07-22): own page, not a 7th tab.** This scope said "wired into a new tab in `dashboard-analytics.html`", which contradicted Phase 4's Developer Note ("a 7th tab would bury exactly the distinction being defended" — Demand Forecast is likewise a Pillar-2 deliverable). ### Phase 7 — Recommendation Engine + Cost/Revenue Estimation + Vehicle Replacement (MVP)
- **Objective:** Deliver forecast-driven Decision Support — the component that consumes both Business Rule Validation (Phase 3) and AI Prediction (Phase 6).
- **Scope — Recommendation Engine pipeline (approved, do not alter):**
  ```
  Historical Dataset
          |
          v
  Demand Forecast
          |
          v
  Candidate Trip Generation   (translate a forecast signal into a concrete route+time-window candidate, same pattern as existing createExtraTrip())
          |
          v
  Recommendation Engine
     +--------------+--------------+
     v              v              v
  Bus Selection  Driver Selection  Cost Estimation
  (reuse findBestAvailableBus()/findBestAvailableDriver() — already rule+availability filtered.
   NOTE: both are private — this step needs an additive public overload on TripService.
   See Developer Notes; do not flip the existing methods to public.)

          |
          v
  Business Rule Validation   (reuse Phase 3 dry-run wrapper — final confirmation gate)
          |
          v
  Final Recommendation Card
  ```
  - If the final Business Rule Validation gate fails (resource became unavailable while the card was pending review), mark the card as **stale — needs regeneration**. Do not silently auto-retry.
  - Vehicle Replacement Recommendation (MVP): independent of the pipeline above. The recommendation is based on existing operational signals available in the current system, primarily the vehicle's lifetime odometer and historical incident frequency. The exact scoring heuristic is intentionally left implementation-defined so it can evolve without introducing new entities. A full version (with a `MaintenanceRecord` entity and richer lifecycle history) remains a stretch goal. See Developer Notes for the signal deliberately excluded, and why.
- **Dependencies:** Phase 3, Phase 6. *(Phase 4 was listed here until 2026-07-17 — a relic of the draft in which Cost/Revenue Estimation still lived in the Phase 4 group before being moved here (see Developer Notes); nothing in this pipeline consumes Phase 4's deliverable.)*

- **Refactoring risk:** Low — everything reuses existing methods; no changes to `TripService` **logic**. It does, however, require one **additive** change to `TripService`: `findBestAvailableBus()`/`findBestAvailableDriver()` are `private`, so the new `RecommendationService` cannot call them. Expose them through new public overloads following the precedent already set by `getDrivingHoursForDate()` (private impl + public overload). No existing method changes visibility; no logic is touched. See Developer Notes.
- **Current status:** **COMPLETE (2026-07-24)** — split into three steps with owner approval, because the phase turned out to hold three separable deliverables with very different risk profiles. **All three steps delivered:** step 1 (Vehicle Replacement) 2026-07-22, step 2 (Recommendation Engine core, revenue) 2026-07-23, step 3 (parametric cost/profit, owner-approved) 2026-07-24.

  - ✅ **Step 2 — Recommendation Engine core** (candidate generation → resource selection → validation gate → card), revenue only. Delivered 2026-07-23. The additive public overloads were added as planned; existing method visibility unchanged and no existing logic touched. See "As delivered (step 2)" below.
  - ✅ **Step 3 — Cost estimation. COMPLETE (2026-07-24).** The blocker was resolved by an owner decision: a **parametric cost model** with **admin-configurable** rates (new `CostParameters` + settings screen), defaulting to sourced market-reference values. Cost + profit now show on each card. See "Blocker → resolution" and "As delivered (step 3)". **Phase 7 is now fully delivered (all 3 steps).**
- **Blocker (2026-07-22) → resolution (owner-approved 2026-07-24): the system holds no cost data, so cost is a *parameter*, not a derivation.** A grep of the whole `domain` package returns **zero** fields for fuel, wages, tolls or per-km cost — operating cost is exogenous to a scheduling schema, so no figure can be derived purely from existing data. Rather than hardcode invented constants (which Section 4's fare/pricing Non-Goal warns against), the owner approved the **canonical parametric approach**: a new admin-configurable `CostParameters` (fuel cost/km, driver wage/hour) whose values are **a business input the operator supplies**, seeded with **illustrative market-reference defaults** (approximate, to be replaced by the operator's real rates / cited at defence). Cost = `fuelCostPerKm × Route.distanceKm + driverWagePerHour × durationHours × requiredDrivers`; profit = `revenue − cost`. This sidesteps the Non-Goal (it accepts *operating-cost parameters* as input, it does **not** build a fare/pricing module) and is defensible precisely because the number's source is the operator, not the developer. It also delivers the first slice of Hidden Cost #5 (business constants become admin-configurable rather than hardcoded). **Still forbidden: hardcoding an opaque cost model with no admin control and no stated source.**

### 2026-07-24 — Hidden Cost #9 fixed: recursive `@Data` `hashCode`/`equals`/`toString` — NOT COMMITTED
- **Not a phase.** A self-contained domain bug fix, taken off the open-owner-rulings list. Scope was held to exactly this defect; no phase work was started.
- **The defect:** every entity carries Lombok `@Data`, which generates `equals`/`hashCode`/`toString` over **every** field. Where two entities reference each other, those methods call each other forever → `StackOverflowError`. The domain has exactly **three** such pairs, confirmed by enumerating every `mappedBy` in `domain/`:
  - `User.driver` ↔ `Driver.user`
  - `Route.routeStations` ↔ `RouteStation.route`
  - `Station.routeStations` ↔ `RouteStation.station`
- **The fix (3 files, annotations + comments only):** `@EqualsAndHashCode.Exclude` + `@ToString.Exclude` on the **inverse (`mappedBy`) side** of all three — `User.driver`, `Route.routeStations`, `Station.routeStations`. Cutting one side breaks the cycle; cutting the inverse side is applied **uniformly** so the rule is one sentence, not three special cases, and it leaves the owning side (the direction code actually reads) fully comparable. No logic, no query, no schema, no template changed.
- **Semantic consequence, accepted:** two `User`s differing *only* in their `Driver` profile now compare equal. Negligible — `id` and `username` are both unique and compared first, and before the fix `hashCode()` on such a `User` **always threw**, so no existing code could have depended on the old behaviour. Confirmed by grep that nothing in `src/main/java` calls `equals`/`hashCode`/`toString` on an entity: every comparison in the codebase is by ID.
- **New test — `EntityEqualsHashCodeCycleTest` (5 tests), three deliberate layers:** (1) hand-wired cycles for all three pairs, which is the strictest form since it does not depend on what Hibernate loads; (2) the real-world shapes — `HashSet<Trip>`, `HashSet<Driver>`, `HashMap<Route,…>`, `.stream().distinct()` — because a bare `hashCode()` call is necessary but not the shape the bug will actually take, and Phase 9 (Customer Portal) is where those get written first; (3) a cycle built by **Hibernate itself** after a normal `findById`, with an `assertSame` guard so the test cannot pass vacuously against a `null` reference.
- **Guard verified non-vacuous:** reverting the three annotations and re-running makes **all 5 tests fail with `StackOverflowError`**; restoring them returns the suite to green. This was checked explicitly rather than assumed — the bug had stayed latent precisely because nothing exercised these methods.
- **Verification:** `mvnw test` **17/17** (was 12/12; the 12 pre-existing tests are unaffected). Working tree confirmed byte-identical after the revert/restore experiment.
- **Two corrections to Register row #9 as originally written:** (a) `originalTrip` was listed as "very likely also affected" — it is **not**. It is set only to an already-persisted, different trip (`TripService:165`), so it forms a finite chain, never a cycle; it was deliberately left inside `equals`/`hashCode`. (b) The row implied the Step 2 equivalence test's native-SQL co-driver insert was forced by this bug. That reason is now gone (the stale comment in `TripServiceAvailabilityContextTest` was corrected), but the native insert was **left as-is** — rewriting a passing test for cosmetics is out of scope.

## 9. Developer Notes

*(Implementation decisions that are not obvious from the code — prevents future sessions from re-litigating the same discussions.)*

- **Why Cost/Revenue Estimation moved out of the "current-state" group (Phase 4) into the post-Forecast group (Phase 7):** "Estimation" implies a forward-looking projection, which needs Demand Forecast's predicted-demand output as input. Driver Recommendation, by contrast, only reports current-day driving-hour workload (`getDrivingHoursForDate()`) and genuinely needs no forecast — it stays in Phase 4.
- **Why Historical Data Preparation (Phase 5) is a separate phase from Demand Forecast (Phase 6):** Bundling backfill script work into "the Forecast phase" blurs an infrastructure/data concern with an algorithm concern. Splitting them makes both easier to grade/present independently in the thesis.
- **Why "Business Rules" and "Resource Availability" are not implemented as two sequential services despite being two boxes in the pipeline diagram:** In the existing code, `TripService.findBestAvailableBus()` already applies both as interleaved filters over the same candidate stream (`.filter(!isBusBusy(...)).filter(!needsMaintenance()).filter(!isNearMaintenance(...))`), and filter intersection is order-independent. Implementing them as two separate, sequential services would duplicate logic that already exists — directly violating the "no parallel architecture" principle. Keep two boxes in diagrams for clarity; implement as one reused call.
- **Why the Recommendation Engine's final Business Rule Validation gate reuses `validateBusForTrip`/`validateStaffForTrip` (via the Phase 3 dry-run wrapper) rather than reusing `findBestAvailableBus`/`findBestAvailableDriver` again:** This mirrors an already-proven pattern in the codebase: `TripService.confirmAutoAssignedTrip()` selects resources via the filtered helper methods, then re-validates via the throw-based validators immediately before committing, because state can change between selection and commit. A Recommendation Card may sit unacted-on for hours or days (much longer than the live AI auto-assign flow), so this staleness risk is even more relevant here — hence the explicit "mark as stale, don't auto-retry" rule if the final gate fails.
- **Why the incident delete-guards deliberately do NOT filter by `IncidentStatus` — a `RESOLVED` incident also blocks deleting its bus (owner-approved 2026-07-17):** `BusService.deleteBus()`/`DriverService.deleteDriver()` call `existsByBusId`/`existsByDriverUserId` with no status filter, so a report closed months ago still prevents a hard delete. This was surfaced by end-to-end verification (a bus whose only incident was `RESOLVED` stayed undeletable) and **explicitly kept**. Rationale: the guard protects **referential integrity**, not open work — an orphaned `incidents.bus_id` is equally broken whether the report is `OPEN` or `RESOLVED`. More importantly, closed reports *are* the data Phase 7's Vehicle Replacement heuristic consumes (incident frequency per bus); silently orphaning them would corrupt that input, defeating the reason `Incident.bus` is mandatory in the first place. Accepted consequences, both deliberate: (a) `Bus` has no soft-delete, so a vehicle with any incident history is retired by moving it to `REPAIRING`, not by deletion; (b) genuinely removing such a bus means deleting its incident reports first, which does destroy that frequency history — an explicit, Admin-made trade rather than a silent one. **Do not "optimise" this to `status IN (OPEN, IN_PROGRESS)`** — that re-opens orphan creation for exactly the closed reports Phase 7 depends on.
- **Why Phase 7 needs additive public overloads on `TripService`, and why that does not contradict "no changes to `TripService`":** Section 3 names five methods to reuse, but as of Phase 2 **only `getDrivingHoursForDate()` is `public`** — `findBestAvailableBus()` (~line 276), `findBestAvailableDriver()` (~335), `validateBusForTrip()` (~684) and `validateStaffForTrip()` (~744) are all `private`. This bites differently per phase, which is why it went unnoticed: Phase 3's dry-run wrappers live **inside** `TripService`, so private access is fine; Phase 4 consumes the one already-public method; but Phase 7's Recommendation Engine is a **new** service (Section 3: new modules follow Controller → Service → Repository), and a separate class cannot call a private method. The resolution is a pattern the codebase already demonstrates: `getDrivingHoursForDate()` exists twice — a `private (Driver, LocalDateTime, Long excludeTripId)` implementation and a `public (Driver, LocalDateTime)` overload delegating to it. Phase 7 adds the same kind of public overload for bus/driver selection: purely additive, no existing method changes visibility, no logic touched. **Do not** instead (a) flip the existing private methods to public wholesale — that widens the blast radius of the most correctness-sensitive code in the app for no benefit, or (b) move the Recommendation Engine inside `TripService` — it is already ~1,100 lines, and Section 3 forbids growing god-components.
- **Why the Phase 3 dry-run wrappers catch only `IllegalArgumentException` (owner-approved 2026-07-20):** that is the exception type both validators use to signal a business-rule violation, and it is the only one that means "this resource is not allowed" rather than "this code is broken". A broader `catch (Exception e)` would convert genuine faults — a `NullPointerException` from a null `bus`, a `LazyInitializationException` from calling the wrapper outside a session, a data-access failure — into a perfectly ordinary-looking `ValidationResult.fail(...)`. Phase 7 would then silently drop a valid bus from a recommendation and report a plausible Vietnamese sentence as the reason, with nothing in the log. Failing loudly is the correct behaviour for a fault; a test locks this in (`busDryRun_nullBus_propagatesNpeInsteadOfFailing`). **Do not widen the catch** to make the API feel more convenient.
- **Why a `ValidationResult` can only ever carry one failure reason, and why that is not a defect (owner-approved 2026-07-20):** both validators are fail-fast — they `throw` at the first rule violated, so a second reason never comes into existence for the wrapper to catch. Reporting all violations at once would mean rewriting them to accumulate into a list, which Phase 3 explicitly forbids (Section 5) and which would touch the most correctness-sensitive code in the app to serve a consumer that does not need it: Phase 7's gate asks only "is this recommendation still valid?", a pass/fail question. If a future requirement genuinely needs complete multi-error reporting, the owner's ruling is that it be treated as a **separate architectural enhancement outside Phase 3**, not as a quiet fix inside the wrappers. A test (`staffDryRun_multipleViolations_reportsOnlyTheFirst`) pins the behaviour so it cannot be "improved" by accident.
- **Why Phase 3 shipped with an integration test when the project's convention is drive-the-app (2026-07-20):** every prior phase delivered a controller and template, so `.claude/skills/verify`'s recipe reached the new code through a real HTTP request. Phase 3 delivers a **service-layer API with no UI** — no endpoint calls the wrappers until Phase 7 exists, so there is literally nothing to drive. An integration test was the only way to execute them at all; the app was still driven separately to prove the throw-based flows unchanged, which *is* reachable. This is not a change of convention: prefer driving the app wherever a phase has a UI surface. The test's most valuable assertion is the **equivalence** one (dry-run's `failureReason` equals the message the throw path raises for the same input) — that is the guarantee the whole phase rests on, and the thing most likely to rot silently if someone later edits one validator.
- **Why `validateBusForTrip`'s warning channel is forwarded into `ValidationResult` rather than dropped:** the method returns a `String` warning that is always `null` today (all maintenance constraints became hard blocks), but the channel is deliberately kept for future non-blocking warnings — see its javadoc. Forwarding it costs one constructor argument and means a future warning automatically reaches Decision Support; dropping it would have created a second place to remember to update.
- **What Phase 7 CANNOT do with a `ValidationResult`, and why (recorded 2026-07-20 so it is not discovered mid-Phase-7):** the failure reason is a **free-text Vietnamese sentence**, not a rule code or enum. Phase 7 can *display* why a Recommendation Card went stale, but it cannot programmatically branch on *which* rule failed — no "if the bus is merely near its maintenance threshold, try the next-best bus; if the driver is over hours, keep the bus and re-pick staff". Giving it that ability would mean adding a code to every `throw` site inside the two validators, i.e. rewriting them, which Section 5 forbids. Two consequences to design around: (a) keep the gate binary — valid, or stale-and-regenerate, exactly as the approved pipeline already says; (b) do **not** parse or `String.contains()` the reason text to recover a category — the messages are user-facing Vietnamese and will change wording. If Phase 7 genuinely needs categorised failures, raise it with the owner as its own scoped change rather than pattern-matching on strings.
- **Why the dry-run wrappers are genuinely side-effect free (verified, not assumed):** "dry run" would be a dishonest name if the validators mutated anything. Checked across both method bodies: no setter, no `save()`, no `delete()`, no collection mutation on a managed entity — the only write is to a local `HashSet` used for duplicate-staff detection. So a dry-run call can be made against a candidate `Trip` (Phase 7) or a persisted one without risk of a stray flush. Re-check this if the validators are ever extended.
- **Why the dry-run methods have new names instead of overloading the validators:** the codebase's precedent for exposing a private method is a same-name public overload (`getDrivingHoursForDate()`). It does not apply here — `validateBusForTrip` already returns `String` and the dry-run must return `ValidationResult`, and Java cannot overload on return type alone with an identical parameter list. Distinct `...DryRun` names were chosen over changing the originals' signatures, which would have meant touching all four throw-based call sites.
- **Why Phase 4 redeclares the 8-hour daily limit instead of extracting a shared constant from `TripService` (decided during implementation, 2026-07-20):** the obvious move was to promote `8.0` to a public constant and reuse it. Grepping all **14** occurrences in `TripService` killed that idea: the literal carries **three different business meanings** that merely share a value — the **daily driving budget** (lines 357, 833, 856, 1024, 1120), the **per-trip share cap** applied when splitting a long trip across co-drivers (343, 502, 817, 1016, 1112), and the **long-trip threshold** that forces an assistant or extra drivers (215, 253, 751, 775). A single `MAX_DRIVING_HOURS_PER_DAY` replacing all of them would be semantically wrong, and would silently couple three independent rules so that changing one changes all three. `DriverRecommendationService.MAX_DAILY_DRIVING_HOURS` therefore mirrors only the **first** meaning, with a comment naming the exact rule (`validateStaffForTrip`'s "vượt 8h/ngày" check) — the same approach `DispatchController` already takes for the 48-hour window. **Do not "clean this up" by extracting one constant** without first separating the three meanings inside `TripService`, which is a much larger, riskier change than it looks.
- **Why `DriverWorkloadDto` was not reused for Phase 4 (the scope line asked that it be considered):** it carries `userId`, `fullName`, `drivingHoursToday` — enough for a ranking cell, not enough for a recommendation. Phase 4 must also show remaining capacity, licence validity + expiry, experience, a status label and a human-readable reason. Widening `DriverWorkloadDto` would have dragged 6 unused fields into `DashboardService`'s already-shipped `topLoadedDrivers` path, i.e. modifying a verified feature to serve a new one. Two DTOs answering two opposite questions is the cheaper, clearer outcome.
- **Why Driver Recommendation lives on its own page rather than as a tab in `dashboard-analytics.html`:** two independent reasons. *Mechanical:* the recommendation is driven by an Admin-chosen date, and `DashboardService.getOverview()` takes no parameters — the entire analytics page is parameterless, so adding a date would have meant changing a shipped signature and recomputing the whole dashboard on every date change. *Conceptual:* Phase 4 is the first **Pillar-2 (Decision Support)** deliverable, and the thesis argument depends on it being visibly distinct from Pillar-1 analytics; a 7th tab would bury exactly the distinction being defended. It still lives under `/admin/analytics/` and is linked from the Drivers tab, so it is inside the analytics area without being inside the analytics page. Converting it to a tab later is cheap if the owner prefers.
- **~~Why the licence check in Phase 4 is "valid today", not "valid on the selected date"~~ — SUPERSEDED 2026-07-20, the rule is now date-aware:** the original note recorded that `Driver.isLicenseValid()` compared against `LocalDate.now()`, so a recommendation for a future date listed drivers whose licence expired before it. It was kept at the time because `validateStaffForTrip()` behaved identically and a divergent second definition would have been worse. **The owner then approved fixing it at the root instead**, which is the outcome that note was pointing at. `Driver` now has `isLicenseValid(LocalDate)` — with the no-arg version delegating to it via `now()`, following the `getDrivingHoursForDate()` precedent — and all six assignment-path checks in `TripService` pass the departure date, while `DriverRecommendationService` passes the Admin's selected date. There is still exactly **one** definition of "valid licence"; it simply takes the date as a parameter now. The no-arg form survives for genuinely now-relative display only (the "Hết hạn" badge on the driver list, which has no trip context). Full record in the Session Log entry for 2026-07-20.
- **Why the Phase 5 backfill guarantees rule-compliance by construction instead of calling the validators (2026-07-20):** the obvious move — push each generated trip through `TripService.createManualTrip()` — does not work, because the validators ask questions about *now*: is this bus `TRAVELING` today, is its odometer near the maintenance threshold today. Neither is meaningful for a trip dated three months ago, and nearly every generated row would be rejected for reasons that never applied on its actual date. `DataInitializer` already establishes that seeding writes through repositories. The generator instead makes violations **impossible to express**: routes are capped at 8 hours, and drivers and buses are rotated by a running index that cannot repeat within a day, so no overlap and no daily-limit breach can occur. This was then *verified against the enforced rules in SQL* (not the construction rule) over the combined dataset — see the Session Log. If the generator is ever changed to allow more trips per day than there are drivers or buses, that guarantee silently disappears; the runner therefore refuses to start in that case rather than producing quietly-invalid history.
- **Why the Phase 5 backfill shifts `odometer` and `lastMaintenanceOdometer` together, and must never be "simplified" to one column (2026-07-20, owner-approved):** the fleet's mileage has to reflect 1,260 completed trips, or Phase 7's Vehicle Replacement heuristic ranks on seeded numbers unrelated to the history it claims to analyse. But crediting `odometer` alone adds ~10,300 km per bus against a 5,000 km `maintenanceThreshold`, which makes `needsMaintenance()` true for the **entire fleet**, `findBestAvailableBus()` reject every vehicle, and dispatch plus AI auto-assign stop working — the backfill would silently brick the operational half of the system. Shifting both columns by the same amount keeps `kmSinceLastMaintenance` (their difference) **unchanged by arithmetic**, so every maintenance rule is a pure function of quantities that did not move. It also happens to be the honest business reading: a bus that covered 10,000 km over twelve weeks was serviced during them. Verified byte-identical across all 19 buses, demo cases included. **Do not "tidy" this into a single-column update.**
- **Why the Phase 5 dataset is idempotent via `(route, departureTime)` rather than a marker column (2026-07-20):** the natural instinct is to add something like `Trip.isBackfilled` so re-runs can be detected. Section 3 explicitly resists growing `Trip`, which already carries FSM, soft-delete and self-reference duties. It turns out no column is needed: every generated row occupies a deterministic slot, so "has this slot been filled?" is answerable with an existence query on data already present. Cost: one derived query. Benefit: re-running the backfill is a no-op instead of a duplication hazard, with no schema change. The random draws are taken *before* the skip check so the pseudo-random sequence does not shift between a first and a later run.
- **Which statuses Phase 6 should count as demand (settled 2026-07-20, so it is not guessed later):** the backfill produces `COMPLETED` (1,183) and `CANCELLED` (77), and **`CANCELLED` rows keep the `ticketsSold` they had before cancellation** — they are not zeroed. That is deliberate: a cancelled trip really did sell those tickets, and zeroing them would destroy information Phase 6 might legitimately want. The recommendation is to forecast on **`COMPLETED` only**, because that is the served-demand series and it is the one whose seasonality and trend were verified. Note this is *narrower* than `DashboardService.OPERATIONAL_STATUSES` (`ACTIVE`/`DEPARTED`/`COMPLETED`), which is a revenue filter, not a history filter — do not reuse that constant here without thinking. Whatever Phase 6 picks, it must **filter explicitly**: silently including `CANCELLED` mixes unserved demand into the actuals.
- **Three off-slot observations Phase 6 must expect (2026-07-20):** grouping `COMPLETED` trips by `(route, HOUR(departureTime))` yields the 15 intended series **plus** three single-observation groups — route 1 @ 21:00, route 2 @ 13:00, route 2 @ 21:00. These are the owner's own pre-existing trips, not backfill artifacts. Phase 6 must not assume every `(route, hour)` group is a full time series; a group with one observation cannot support a moving average or a regression. Either restrict the forecast to the three canonical slots (06/12/18) or require a minimum observation count per series. Do **not** delete the owner's rows to tidy the grouping.
- **Why What-if Simulation is scoped to aggregate/statistical output only:** `findBestAvailableBus()`/`findBestAvailableDriver()` are tightly coupled to live repository queries (not pure functions over an injectable resource pool). Refactoring them to support hypothetical "what if we had N more buses" scenarios cleanly would be the single highest-risk refactor identified across the entire roadmap, touching the most correctness-sensitive code in the app. Rejected in favor of a narrower simulation computed from existing Forecast/Analytics aggregates.
- **Why Vehicle Replacement Recommendation's MVP requires no new entity:** `Bus` only stores a maintenance *snapshot* (current odometer, threshold) — there is no historical log of past maintenance events (date, cost). A full version needs a new `MaintenanceRecord` entity; deferred as a stretch goal so MVP can ship with a simple heuristic on existing fields.
- **Why Phase 4's scope is under-utilization/availability, and deliberately not a "busiest drivers" report (owner-approved 2026-07-17):** the busiest-today ranking already exists — `DashboardService.buildDriverStats()` maps active drivers through `tripService.getDrivingHoursForDate(d, now)` into `DriverWorkloadDto`, sorts descending, and `dashboard-analytics.html` renders it as `topLoadedDrivers`. Phase 4 as originally worded ("wrap `getDrivingHoursForDate()` into a driver workload report") described that shipped feature almost verbatim — the same re-delivery trap caught in Phase 7's Vehicle Replacement scope on 2026-07-17 (see the odometer note below). What is genuinely missing, and what makes Phase 4 a *Decision Support* deliverable rather than a monitoring stat, is the inverse question: *"who has capacity to take a trip?"* — least-loaded active drivers, on a date the Admin picks, framed as a staffing recommendation. Do not "simplify" Phase 4 back into a top-busy list.
- **Why Vehicle Replacement ranks on lifetime `Bus.odometer` + incident frequency, and deliberately not on `Bus.kmSinceLastMaintenance`:** that field is `odometer - lastMaintenanceOdometer`, so it **resets at every service** — a bus with 800,000 km and 50 past services reads a low value. It answers *"is this bus due for a service?"*, not *"is this bus worn out?"*. The first question is **already delivered**: `DashboardService.buildMaintenanceAlerts()` filters `needsMaintenance() || isNearMaintenance(0)` and ranks by `kmSinceLastMaintenance` desc into `BusAlertDto` — the original proposal's I.3.a ("Cảnh báo bảo dưỡng"), live on the dashboard today. Scoring replacement on the same field would re-deliver a shipped feature under a new name, violating the "do not duplicate existing business logic" principle. `odometer` never resets (a true wear/age proxy) and incident frequency is the reliability signal — which is also what finally makes mandatory `Incident.bus` pay off (see the note above on why `bus` is mandatory). Do not "simplify" this back to `kmSinceLastMaintenance`.
- **Why Phase 4 was removed from Phase 7's dependencies (owner-approved 2026-07-17):** the line `Dependencies: Phase 3, Phase 4, Phase 6` had been present verbatim since the roadmap's first commit (`79b5c4a`). It dates from the draft in which Cost/Revenue Estimation still lived in the Phase 4 "current-state" group — back then Phase 7 genuinely consumed a Phase 4 deliverable. When Cost Estimation moved into Phase 7 (see the first note in this section), the dependency became a relic: the approved pipeline consumes only Phase 3's dry-run gate and Phase 6's forecast, and Phase 4's under-utilization report feeds nothing in it. The roadmap's own convention (Phase 9: *"Technically independent … but deliberately scheduled last"*) is to state ordering preferences as prose, not as dependencies — so the relic was removed rather than reworded. Phase 4 remains scheduled before Phase 7 simply by its number.
- **Why Demand Forecast keys on `Trip.departureTime`, and deliberately not on `Trip.createdAt` (owner-approved 2026-07-17):** two independent reasons. *Semantics:* demand for a route/time-slot means tickets sold on trips **departing** in that slot — `departureTime` is the business time axis, and every existing time-window query in `TripRepository` (dispatch board, daily driving-hours, analytics ranges) already keys on it; `createdAt` records when the row was inserted, which for admin-created trips is an operational accident. *Mechanics:* the Phase 5 backfill cannot backdate `createdAt` anyway — Hibernate's `@CreationTimestamp` generates the value at INSERT regardless of anything assigned, `updatable = false` blocks later JPA updates, and the entity javadoc explicitly says it is never set manually. **Do not "solve" this with a native-SQL backdate** — that would falsify a technical audit timestamp to serve a purpose `departureTime` already serves honestly. `createdAt`'s intended consumer remains the booking-velocity stretch goal (Hidden Cost #2), where insert-time is exactly the correct meaning.
- **Why `ddl-auto=create-drop` must be addressed before Phase 5/6:** `DataInitializer.run()` calls `deleteAll()` on every repository and reseeds fixed demo data on every application startup. Any backfilled historical dataset would silently be destroyed on the next restart, and Demand Forecast would then run on near-empty data **without raising any error** — the failure mode is silent and easy to miss, not a crash. *(Resolved in Phase 0.)*
- **Why persist became the default and wiping became opt-in (Phase 0, owner-approved):** the two options were "keep wipe-on-start as default, persist via a flag" versus "persist by default, wipe via a flag". The second was chosen because the failure modes are asymmetric — forgetting the flag under a wipe-default silently destroys the historical dataset (unrecoverable, no error raised), whereas forgetting the flag under a persist-default merely means you don't get fresh sample data (obvious and trivially fixed by re-running with `demo`). The cost accepted: a fresh clone starts empty and must be seeded once with `-Dspring-boot.run.profiles=demo`, and the old "restart = clean data" habit no longer applies.
- **Why gating `DataInitializer` with `@Profile("demo")` was the actual fix, not the `ddl-auto` change:** `DataInitializer implements CommandLineRunner`, so it runs on *every* startup independently of Hibernate's DDL policy, and its first action is `deleteAll()` across all repositories. Setting `ddl-auto=update` alone would have preserved the *schema* while the bean still deleted every *row* — the data loss would have persisted while looking fixed.
- **Why `@CreationTimestamp` was chosen over Spring Data JPA Auditing for `Trip.createdAt`:** the `Trip` entity already relies on Hibernate-specific annotations (`@SQLDelete`, `@SQLRestriction`), so `@CreationTimestamp` matches the established style and requires no extra configuration. Spring Data Auditing would have needed an `@EnableJpaAuditing` config bean plus `@EntityListeners(AuditingEntityListener.class)` on the entity — new machinery for a single field, contradicting the "minimize refactoring / no new abstractions unless required" principle.
- **Why `Trip.createdAt` is nullable rather than `NOT NULL`:** `@CreationTimestamp` only populates on INSERT, so rows that already existed when the column was introduced (via `ddl-auto=update`'s `ALTER TABLE`) hold `NULL`. Enforcing `NOT NULL` would either break the schema update on a populated database or demand a backfill migration that no tooling exists for. Any future consumer of this field must handle `NULL`.
- **Why test isolation (`src/test/resources/application.properties`) was folded into Phase 0:** analysis revealed that with no test-specific config, `BusManagementApplicationTests` loaded the app's own `application.properties` and therefore dropped and reseeded the **real** `busmanagement` database on every `mvnw test` run. This is the same root cause as the phase objective (unintended data destruction), so fixing it separately would have left Phase 0's guarantee incomplete — a single test run would still wipe the Phase 5 dataset. Note that `src/test/resources/application.properties` **shadows** the main file entirely (test classpath precedence; the files are not merged), which is why every property is redeclared there rather than only the overrides.
- **Why Driver creation builds the `User` and the `Driver` in one form (Phase 1, owner-approved):** `Driver` maps its PK to `User.id` via `@MapsId`, so a `Driver` row cannot exist without a `User` row. The alternative was a two-step flow (create the `User` at `/admin/users/new`, then attach a driver profile), which keeps the two concerns separate but costs two screens. The combined form was chosen because it mirrors what `DataInitializer.createDriver()` already does and treats "a driver" as one thing in the UI. Consequence: `DriverService.deleteDriver()` deletes the **`User`** and relies on `User.driver`'s `cascade = ALL` to remove the `Driver` — deleting only the `Driver` would strand an unreachable `ROLE_DRIVER` user, since the combined form offers no way to re-attach a profile to an existing account.
- **Why the driver-deactivation guard reuses `TripService`'s `busyStatuses` set:** the question "which trip statuses mean a driver still has obligations?" was already answered in the codebase by `TripService.isDriverBusyInWindow()`, which uses `{PENDING_APPROVAL, ACTIVE, DEPARTED}`. `DriverService.BUSY_STATUSES` repeats that exact set instead of inventing a second definition of "busy". The guard itself mirrors `BusService.saveBus()`, which already blocks moving a bus to `REPAIRING` while it holds unfinished trips.
- **Why `monthlyRestDays` and `totalDrivingHours24h` are hidden from the driver form (owner-approved):** `monthlyRestDays` is read nowhere in the codebase — the original proposal's "nghỉ đủ 2 ngày/tháng" rule was never implemented — so exposing an editable field would imply an effect that does not exist. `totalDrivingHours24h` *does* have an effect (`getDrivingHoursForDate()` adds it to the daily total) but is documented in `TripService` as mock seed data destined to be fed by IoT/GPS, so it is not an admin-editable field either. Both were left out rather than shipped as misleading inputs.
- **Why editing a route's stops deletes and rebuilds the `RouteStation` rows instead of updating them:** `RouteStation`'s PK is the composite `(routeId, stationId)`, so "changing which station a stop points at" is not an update — it is a different primary key. `Route.routeStations` also declares `cascade = ALL` **without** `orphanRemoval`, so clearing the collection would not delete anything. `RouteService.saveRoute()` therefore deletes the existing rows explicitly (then `flush()`es, so the re-inserted rows cannot collide with the old ones inside the same transaction) and re-creates them with `stopOrder` renumbered from the submitted order.
- **Why a route cannot revisit the same station (e.g. A→B→A):** the composite PK `(routeId, stationId)` makes a repeated station impossible at the schema level. `RouteService.validateRoute()` rejects duplicates up front with a clear business message (and the form pre-checks in JS) rather than letting it surface as a PK violation. This is a known modelling limitation, not a validation choice — round-trip routes need a different `RouteStation` key design.
- **Why the Dispatch Center repeats the 48-hour window instead of importing it:** `DashboardService.UPCOMING_TRIPS_WINDOW_HOURS` is `private`, and widening its visibility purely for reuse would couple the dispatch screen to the analytics service for a single integer. `DispatchController` declares its own constant with a comment pointing at the Dashboard one, so the two stay recognisably the same concept without a structural dependency. If a third consumer appears, promote it to a shared constant then.
- **Why `findDispatchBoardTrips` JOIN FETCHes only `t.coDrivers` and not `r.routeStations`:** Hibernate rejects fetching two `List` (bag) collections in one query (`MultipleBagFetchException`) — the same constraint documented on `Route.routeStations`, which uses `@BatchSize(20)` for that reason. The dispatch board needs bus/driver/assistant eagerly (it renders them), and gets departure/destination names through the batched `routeStations`.
- **Why `Incident.status` has no FSM even though `Trip.status` does (Phase 2, owner-approved):** the two model different things. A trip's status drives real side effects (bus availability, odometer, ticket sales) so an illegal transition is a data-integrity problem — hence the whitelist FSM. An incident's status is a human progress note with one derived field (`resolvedAt`); closing one by mistake is an ordinary clerical error, and an FSM making `RESOLVED` terminal would force the Admin to delete and re-file the report to correct it. Free transitions plus automatic `resolvedAt` sync was chosen instead. Do not "harmonise" this with the Trip FSM without asking.
- **Why `Incident.bus` is mandatory while `trip` and `driver` are optional (Phase 2, owner-approved):** every incident is ultimately about a vehicle, and a mandatory `bus` is what makes "which bus fails most often" answerable — the intended input for Vehicle Replacement Recommendation in Phase 7. `trip` must stay optional because a bus can break down in the depot, outside any trip; `driver` must stay optional because there is no authentication (`SecurityConfig` is permit-all and `02_project_context.md` forbids building on `SecurityContextHolder`), so the reporter cannot be derived from a session and the Admin may simply not know it.
- **What happens when a trip referenced by an incident is soft-deleted (probed, Phase 2):** `Trip` carries `@SQLRestriction("is_deleted = false")`, so a soft-deleted trip disappears from every query — including the `LEFT JOIN FETCH i.trip` in `IncidentRepository`. Verified behaviour: nothing throws (no `EntityNotFoundException`), the incident list and edit pages still load, and the incident simply renders as "Ngoài hành trình" as if it never had a trip. Two consequences to be aware of: the display is silently misleading in that state, and re-saving such an incident writes `trip_id = NULL` permanently, because the soft-deleted trip is absent from the form's dropdown. This was accepted rather than fixed — the alternatives (blocking trip deletion when incidents reference it, or bypassing the restriction) would both reach into `TripService`, which Phase 2 is explicitly not allowed to do.
- **Why a separate MySQL test database instead of an in-memory H2:** H2 would have required a new dependency and introduced dialect divergence from production behaviour (the JDBC URL uses the MySQL-specific `sessionVariables=foreign_key_checks=0`, and `@SQLDelete` issues raw MySQL SQL). Pointing tests at `busmanagement_test` with `createDatabaseIfNotExist=true` keeps the dialect identical, adds no dependency, and needs no manual `CREATE DATABASE` step.
- **Why Booking's seat-decrement needs pessimistic locking:** The original thesis proposal (`.agent_instructions/03_functional_spec.md`, section IV.2) explicitly called out the concurrent double-booking race condition as a requirement to handle. `Trip.ticketsSold` today has zero concurrency protection because only a single Admin edits it manually; once real customer bookings exist, this becomes a genuine multi-writer field.
- **Why `AdminBusController` (not `AdminTripManagementController`) is the required template for new controllers:** `AdminTripManagementController` injects `TripRepository`/`RouteRepository`/`BusRepository`/`DriverRepository` directly alongside `TripService` — a known, already-documented layering violation (`docs/reports/project_report.md`, Warn #4). New controllers in Phase 1/2 must not copy this pattern.
- **Why Demand Forecast predicts the occupancy RATE and not `ticketsSold` (decided 2026-07-22; the first proposal was `ticketsSold` and was reversed):** the statistical argument is real but secondary — `ticketsSold = occupancy × the capacity of whichever bus was rotated in`, the fleet holds three capacities (22/40/50) which **all appear in all 15 series**, and the measured coefficient of variation is 0.32–0.35 for `ticketsSold` against 0.08–0.11 for occupancy, while mean occupancy by capacity is 0.700/0.705/0.702. So most of `ticketsSold`'s movement is which bus was assigned, not whether passengers came. **The decisive argument is architectural.** The system already has exactly one definition of "this trip is running hot": `Trip.getOccupancyRate() > 0.90` (`Trip.needsReinforcement()`), and it already drives the whole extra-trip flow in `TripService.scanAndSuggestExtraTrips()`; `DashboardService` even annotates its copy of the constant with *"không định nghĩa lại 'hot trip'"*. Forecasting in occupancy means Phase 7's Candidate Trip Generation reuses that threshold with a predicted value substituted for an actual one. Forecasting in tickets would force Phase 7 to invent a second threshold in units the codebase has never used — the exact duplication Section 3 forbids. Absolute passenger counts remain available downstream: multiply the forecast by the capacity of the bus that Bus Selection actually picks, which happens *after* candidate generation in the approved pipeline anyway. **Do not "improve" this into a ticket-count forecast.**
- **Why Demand Forecast got its own page although Phase 6's scope line said "a new tab" (owner-approved 2026-07-22):** the roadmap contradicted itself. Phase 6's scope says *"wired into a new tab in the existing `dashboard-analytics.html`"*, while Phase 4's Developer Note argues that a Pillar-2 (Decision Support) deliverable must stay visibly distinct from Pillar-1 analytics because *"a 7th tab would bury exactly the distinction being defended"* — and Demand Forecast is equally Pillar-2. Note the two reasons Phase 4 gave are separable: the **mechanical** one (`getOverview()` takes no parameters) genuinely does not apply to Phase 6, which needs no parameters and could have been a tab; only the **conceptual** one carries over. The owner was given both readings and chose the own page, so `/admin/analytics/forecast` follows the Phase 4 shape: own controller, own template, under `/admin/analytics/`, linked from the Analytics "Lấp đầy" tab. The existing `tab-ai` was considered as a counter-precedent and rejected — it reports *counts of past AI suggestions*, which is descriptive analytics about a feature, not a predictive output.
- **Why the day-of-week index is pooled across all series instead of fitted per series:** each series holds ~78 observations, which is ~11 per weekday — enough to be noisy, not enough to be stable, and a wrong weekday factor multiplies straight into every prediction for that day. Pooling over all qualifying observations gives ~170 per weekday. This is also the honest reading of the phenomenon: a weekend travel peak is a property of the calendar, not of one route at one hour. If a route ever genuinely has its own weekly shape (a commuter route, say), that is a real modelling change to raise with the owner, not something to slip in by switching the grouping.
- **Why the backtest recomputes the seasonal index from the training window only:** the obvious implementation passes the already-computed global factors into the backtest, and it is wrong — those factors were fitted on all data including the 14 held-out days, so information from the test window leaks into the model and the reported error comes out better than the model deserves. The reported 4.26% would have been a number nobody could defend. `backtest()` therefore builds its own factors from the training pool. **Do not refactor this to reuse the outer factors "to avoid duplicate work"** — the duplicate work is the entire point.
- **Why the moving average is a 7-calendar-day window and not "the last 7 observations":** ~6% of trips are `CANCELLED` and excluded, so each series is missing 2–8 days out of 84. "The last 7 observations" would therefore span more than 7 calendar days and cover the weekdays unevenly — re-contaminating the average with exactly the weekly cycle a 7-day average exists to remove, and doing so by a different amount for every series. The window is expressed in dates, and a fully-empty window falls back to the series mean rather than 0 (returning 0 would assert "nobody travelled", which is a claim, not a missing value).
- **Why `ForecastService` redeclares the 0.90 reinforcement threshold instead of calling `Trip.needsReinforcement()`:** there is no `Trip` entity to call it on — the value being tested is a predicted number for a trip that does not exist yet, and the service deliberately reads projections rather than entities (see Hidden Cost #4). This follows the precedent already set by `DashboardService.HOT_TRIP_OCCUPANCY_THRESHOLD`, which redeclares the same constant with a comment pointing back at the domain method, and by `DriverRecommendationService.MAX_DAILY_DRIVING_HOURS`. If the threshold in `Trip` ever changes, all three must change together.
- **Why Vehicle Replacement counts only two of the five incident types (2026-07-22):** `IncidentType` has five values, but only `VEHICLE_BREAKDOWN` (technical failure) and `ACCIDENT` (physical damage to the vehicle) say anything about the *vehicle*. `ROAD_ISSUE` is traffic, flooding or a closed road; `STAFF_ISSUE` is a driver falling ill; `OTHER` is unknown by definition. Counting those against a bus means a healthy vehicle loses points because it once sat in a traffic jam — the incident belongs to the journey, not the machine. Worth knowing when reading the code: **at the time this shipped, all 8 incidents in the database were of the two counted types, so the filter changed no output at all.** It is there for the first `ROAD_ISSUE` row anyone records, and removing it "because it makes no difference" would silently corrupt the score from that moment on. If `OTHER` ever needs to count, that is an owner decision about what `OTHER` means, not a tweak.
- **Why the replacement score is relative (min-max across the fleet) and is presented as a priority order rather than a verdict (2026-07-22):** the tempting alternative is an absolute rule — "over 500,000 km, replace it". There is nothing in the system to justify a number like that: `Bus` has no manufacture year, no purchase date, no service-cost history (see the note below on why no new entity), so any threshold would be invented, which Section 4's "never invent business rules" forbids. Min-max normalisation against the current fleet is honest about what it is: a **ranking**, not a diagnosis. The known consequence, stated on the screen itself: the top bus always scores 100 even if the entire fleet is nearly new, so the score must never be read as "this vehicle is worn out". This is also why the raw odometer and raw incident counts sit next to the score in their own columns — the Admin is meant to check the ranking against reality, not trust the number.
- **Why the weighting is 70% wear / 30% reliability, and what should change it (2026-07-22):** the weights encode **signal quality**, not importance. The odometer is continuous, present on every bus, and separates all 20 of them; incident counts are sparse, integer-valued, and as of shipping 14 of 20 buses sit tied at zero. Weighting a signal that cannot separate the majority any higher would just add noise. The service therefore computes and publishes its own thin-signal indicator (`busesWithoutIncidents * 2 >= fleetSize`) and the page renders an explicit warning when it trips, so nobody reads the output as a balanced two-factor model. **When real incident data accumulates, this weighting is the first thing to revisit** — and the indicator will stop firing on its own, which is the signal to do it.
- **Why Vehicle Replacement needs no new entity, and what the stretch version would need:** `Bus` stores only a maintenance *snapshot* (current odometer, last-service odometer, threshold) — there is no log of past service events, their dates or their cost, so "how expensive has this vehicle been to keep running?" is unanswerable today. The MVP therefore ranks on the two signals that *do* exist. A full version needs a `MaintenanceRecord` entity (date, cost, work done) which would make total cost of ownership computable; that remains a stretch goal, deliberately outside Phase 7 so the MVP could ship without a schema change.
- **Why Phase 6 reads through a projection query rather than loading `Trip` entities:** Hidden Cost #4 has two halves — the unpaginated payload *and* in-memory Java-stream aggregation over entities in `DashboardService`. A forecast over 1,183 rows written in the `DashboardService` idiom would have joined the second half of that problem on a page that reloads it on every view. `TripRepository.findDemandHistoryByStatus` returns 4 scalar columns, following the `findSeatsAndSoldByStatus` precedent, and the page measures **0.10 s / 37 KB**. The trade-off accepted: with no entity in hand, the occupancy division is written out in the service instead of calling `Trip.getOccupancyRate()` — the third place in the codebase to do that arithmetic (`buildOccupancyStats()` is the second). It is commented as such. Prefer the projection; do not "clean this up" by loading entities to reuse the getter.
- **Why the Recommendation card is recompute-per-view and creates nothing (Phase 7 step 2, owner-approved 2026-07-22):** the roadmap pipeline speaks of a "card pending review for hours/days", which reads like a persisted artifact. The owner chose the lighter model: the screen recomputes on every open, writes nothing, creates no trip — identical in spirit to the other two Pillar-2 screens (Driver Recommendation, Vehicle Replacement), which also "assign nobody". Two consequences to design around, both accepted: (a) there is no real pending window, so `RecommendationStatus.STALE` is essentially unreachable here — selection and the final gate run microseconds apart on the same read; it is kept anyway as an honest outcome and as a tripwire if the selection and validation logics ever diverge; (b) making a recommendation real is done by the Admin through the existing create/approve flow, not by this screen. If a persisted card lifecycle is ever wanted, that is a new entity and an owner decision, not a quiet addition.
- **Why revenue uses the latest historical price per (route, hour), and why that is a repository projection not a `Trip` load (Phase 7 step 2):** a forecast candidate is a `(route, date, hour)` with no `Trip`, so it has no price of its own. Section 4 says `Trip.price` is consumed **as-is**, so the engine reuses a real historical price rather than inventing one: the most recent `COMPLETED` trip's price for that `(route, departureHour)`, via `TripRepository.findPriceHistoryByStatus` (3 scalar columns, ordered by departure desc; the service keeps the first seen per slot). This follows the Hidden Cost #4 lesson — a business-oriented projection, never a `List<Trip>` when only the price is needed. Absolute revenue is then `round(forecast occupancy × the capacity of the bus Bus Selection actually picked) × that price`, which is exactly the "multiply the occupancy forecast by the chosen bus's capacity" path the Phase 6 occupancy-vs-tickets note left open.
- **Why the Recommendation engine reads `needsReinforcement` and never the 0.90 constant (Phase 7 step 2):** the "is this hot?" decision is already made inside the forecast and baked into `ForecastPointDto.needsReinforcement` (which `ForecastService` computes against its own redeclared threshold, itself pointing back at `Trip.needsReinforcement()`). The engine consumes that boolean, so 0.90 stays in exactly its three existing homes (`Trip`, `DashboardService`, `ForecastService`) and no fourth copy is created. This is the cleanest possible reading of the roadmap's "reuse the > 0.90 threshold instead of inventing a second definition of hot trip" — reuse the *decision*, not the *number*.
- **Why the availability preload (`AvailabilityContext`) is a data-only class and the rules stayed in `TripService` (Phase 7 step 2, owner-approved 2026-07-23 — the "safe" option):** reusing `findBestAvailableDriver` once per candidate cost ~18.7 s because each call re-queries per driver (hours + busy) — Hidden Cost #4 territory on a Decision-Support page. The owner approved an optimisation **on condition that no second copy of the business rules is created**. Resolution: `AvailabilityContext` holds only the preloaded schedule (one query for the whole horizon); `findBestAvailableBus/Driver` and the three data leaves (`isBusBusy`, `isDriverBusyInWindow`, `getDrivingHoursForDate`) gained an **additive `ctx` overload** — `ctx == null` runs the original SQL (the dispatch/assignment path passes `null` and is byte-for-byte unchanged), `ctx != null` evaluates the *same* rules over preloaded data. The hour rule was extracted into one `sumDrivingHours(...)`; the interval-overlap and role-membership predicates are foundational Java operations that mirror the JPQL, **accepted by the owner as primitives, not a second rule-set, precisely because a test pins their equivalence** (`TripServiceAvailabilityContextTest`, boundary cases). Result: ~0.97 s (≈19×), output byte-identical. **Do not** widen this into a general "load everything and filter in Java" pattern for the live path — that full unification is a separate, owner-gated refactor phase, deliberately not folded into step 2.
- **Why the final validation gate runs on SQL while selection runs on the context (Phase 7 step 2 — intentional, not an inconsistency):** the gate (`validateBusForTripDryRun`/`validateStaffForTripDryRun`) has no `ctx` overload and hits the live DB. That is deliberate: it mirrors `confirmAutoAssignedTrip` (select via filtered helpers, then re-validate independently before committing). Selection-via-context is the fast path; the gate is an independent SQL re-confirmation of it, so if the context ever missed a trip (a preload-span bug), the gate would catch it and mark the card STALE rather than emit a false recommendation. The per-card gate cost is a small fixed number of queries (not N-drivers), which is why the page is still ~1 s.
- **Latent recursive-`hashCode` bug on `@Data` entities — found in Phase 7 step 2, ~~NOT fixed~~ FIXED 2026-07-24 (Hidden Cost #9):** `User` and `Driver` are `@Data` and reference each other, so `hashCode`/`equals`/`toString` recurse forever; flushing a `@ManyToMany` co-driver throws `StackOverflowError`. It has stayed latent only because every route is ≤ 8h (so `requiredDrivers` is always 1 and no co-driver is ever persisted), and the read-only Recommendation path never flushes such a collection. The fix (annotate the back-references with `@EqualsAndHashCode.Exclude`/`@ToString.Exclude`, or drop `@Data`) changes entity equality semantics and is therefore its own owner-decided change, kept out of step 2. The step-2 equivalence test works around it by inserting the co-driver join row with native SQL. ~~**Do not persist a co-driver trip in a test or seed without addressing this first.**~~ **UPDATE 2026-07-24 — this is now done; the warning above is lifted.** All three bidirectional relations are cut on the `mappedBy` side and pinned by `EntityEqualsHashCodeCycleTest`. See Register row #9 and the 2026-07-24 Session Log entry.
- **The entity `equals`/`hashCode`/`toString` convention: ID-based, uniform across all entities (2026-07-24, owner-approved — supersedes the earlier "cut the `mappedBy` side" note).** *History:* the first fix (commit `a4a3bcf`) only excluded the three bidirectional back-references, which stopped the recursion but left `hashCode`/`toString` still walking LAZY associations (extra queries / `LazyInitializationException` when detached) **and** left `equals`/`hashCode` depending on mutable fields. The owner then approved the canonical JPA pattern, which fixes all of that at the root:
  - **The rule (one sentence):** an entity's `equals`/`hashCode` is its `@Id` and nothing else — `@EqualsAndHashCode(onlyExplicitlyIncluded = true)` at the class + `@EqualsAndHashCode.Include` on the id field — and **every** association is `@ToString.Exclude`d.
  - **Why it is correct, not just convenient:** two rows are equal iff they are the same DB row, which is exactly how every comparison in `src/main/java` already works (all by id). A method reading only a scalar id **cannot** recurse and **cannot** lazy-load. And because the id is stable while the row's other columns are not, `hashCode` no longer changes when e.g. `ticketsSold` is updated — an entity put in a `HashSet` stays findable. Field-based `@Data` `hashCode` violated this.
  - **The one edge it does *not* cover (accepted):** a **transient** entity has `id == null`, so two unsaved entities compare equal and an entity's hashCode changes across the initial save. The codebase never puts unsaved entities in hash collections (Phase 9 search results are always persisted rows), so this is documented, not defended. The by-the-book maximum — a hand-written constant `hashCode()` — was **declined** to keep the all-Lombok style; revisit only if a real transient-in-`Set` use case appears.
  - **`RouteStationId`** stays `@Data` field-based on purpose: it is an `@Embeddable` composite-key **value object**, where value equality (both ids) is the correct contract.
  - **For anyone adding an entity (e.g. `Booking` in Phase 9):** add `@EqualsAndHashCode(onlyExplicitlyIncluded = true)` + `@EqualsAndHashCode.Include` on its id, `@ToString.Exclude` on every association, and a case to `EntityEqualsHashCodeCycleTest`. Nothing else in the codebase will catch a miss — the failure surfaces only in the first `Set<entity>`/entity-keyed map/`distinct()` someone writes, most likely in Phase 9.
  - **Still open under Hidden Cost #4 (not this fix):** `open-in-view=true`, no pagination, entity-stream aggregation. Reading a LAZY association *elsewhere* (a template, a service) outside a session can still throw — id-based `equals`/`hashCode` removes that risk **from those three methods**, not from the whole app.
- **Why cost is a parameter, not a derivation, and the one rule to remember (Phase 7 step 3, owner-approved 2026-07-24):** operating cost (fuel, wages, tolls) is **exogenous** to a scheduling schema — there is literally no field to derive it from, so no amount of cleverness turns existing data into a cost figure. The choice is only *where the number comes from*. Hardcoding an invented constant would quietly build the fare/pricing model Section 4 lists as a Non-Goal; the accepted answer is `CostParameters` — the rate is a **business input the operator supplies**, defaulting to sourced market references. This is defensible precisely because the source is the operator, not the developer. `cost = fuelCostPerKm × distanceKm + driverWagePerHour × durationHours × requiredDrivers`; `profit = revenue − cost` (may be negative — show it, do not clamp). **The rule to remember:** the recommendation screen is recompute-per-view and **persists nothing**, so editing a rate is always safe. **But if any future phase persists a computed profit/cost** (a saved Phase 8 scenario, a persisted recommendation card, an export), it MUST **snapshot the rates used at compute time** alongside the figure — otherwise the stored number silently drifts the next time an admin edits the rate, and a "3.2M profit" recorded last week no longer matches its own inputs. Recompute-per-view is what makes today's design immune; persistence removes that immunity.
